CocoaPods全解析

安装CocoaPods

cocoaPos和packageManager是最常用的iOS组件化管理工具。其中cocoaPods以Ruby作为源开发的一款工具，可以通过Gem或homebrew进行安装。MacOS自带Gem工具，通常情况下可以直接用Gem进行安装和本地管理。

  1. Gem Search用于搜索远端的库。
    gem search cocoaPods
  2. Gem Install用于安装所需的库，可以用--version指定版本号。
    gem install cocoaPods, gem install cocoaPods --version=1.3.1
  3. Gem List用于陈列本地已安装库，该命令会罗列附属的cocoaPods库及其版本号。
    gem list cocoaPods
  4. Gem Uninstall用于卸载本地已安装库，同样可以--version指定版本。
    gem uninstall cocoaPods， gem uninstall cocoaPods --version=1.2.1
  5. Gem Update用于进行更新，该命令不会自动卸载旧版本。
    gem update cocoaPods

通过Gem可以搜索到远端有许多cocoaPods的插件，诸如cocoapods-packager和cocoapods-trunk(前者可以命令行单步生成.framework，后者可以单步生成.a)，由于Xcode更新等原因，插件的管理有时候需要手动执行。

创建Pods索引

CocoaPods索引目录以Git仓库托管，所以创建私有目录要求绑定远端的一个.git库。完成绑定后，无论远端的文件层级结构是什么，cocoaPods都以一级目录作为索引目录。

  1. 创建一个用于管理所有CocoaPods库的Git库。该步原理完全参照git原理，remote可以为本地路径，该路径不存在或访问级别受限可能会导致部分pod命令无法执行。
    git create MyPodSpec

  2. pod创建本地索引并指向该remote仓库。

  1. 创建成功后，可以在~/.cocoapods/repos/目录下检索到本地库。该目录本身即为单纯的.git仓库，当pod命令遇到问题时可以考虑以git命令强制更新。

Pod仓库管理

Pod repo命令

除了常用操控pod project的命令外，pod还包含pod repo命令用于直接操控pod本地仓库。

Pod cache命令

Pod会在仓库附属文件夹下建立缓存pod cache，缓存的管理需要用户手动进行。

运作原理

分析以下两条pod常用命令的异同

pod install
pod update --no-repo-update

install与update的区别在于是否就组件包进行大版本的更替，其中install只会对形如1.1.1.1最后一位的数值变动进行更新，而update会尽可能地更新至最新版本，两者的区别会在项目管理中详述。
在版本号已为最新的情况下，两者的运作效果是完全一致的。
当执行pod命令时，cocoaPods会顺序执行以下的最长动作链。

  1. pod repo update 更新仓库
    • -> 若执行的是install或–no-repo-update则跳过此步
    • cd [.cocoaPods path]
    • git pull origin master
  2. pod check PodLock 检查比对podfile与podlock文件中的版本号
    • -> 若执行的是install并且lock一致则跳出
    • -> 若版本号已为最新则跳出
  3. pod cache search 查询缓存
    • -> 若cache中存在最新版本的cache，则直接使用本地缓存的文件，该情况有可能导致，当远端不更新版本号而是tag了一份新代码的时候，本地pod只检查缓存而不检查缓存和云端是否一致。
  4. pod repo search 查询索引
    • -> 若索引不存在则跳出
    • 查询索引时永远只会检查本地的.cocoapods目录，因此若在中途手动修改了.cocoapods文件夹，pod命令依然会按照本地的状况执行，该策略可用于当某一.podspec文件有误而云端未及时更新(或单纯进行本地测试时)，仅需要手动将.podspec文件更新至索引目录即可完美执行pod命令。
  5. git 安装
    • git clone [project name] -> 当检索到有需要安装的新项目时，会在cache中下载一份项目代码，而后按照从本地索引.podspec文件中描述的策略，安装到XcodeProject下。项目扩展原理完全借助了iOS提供的XCBuild命令，原则上cocoaPods工具只是.podspec描述文件里的配置项转换成了XCBuild命令并shell执行。

了解原理后我们可以发现，在该顺序过程中，每一步理论上都可以跨越cocoaPods的机制手动执行，在真实环境中可以适度改变索引、仓库、缓存等以达到强制执行的效果。
注：手动变更而未保存的.cocoapods目录无法通过pod update也是因为git要求目录下所有文件stage后才能进行pull操作。

Pod项目管理

Step.1 - 创建Pod工程

pod项目可以通过pod直接进行建议的空模版创建

pod lib create [项目名]
过程中有默认选项需要用户选择

也可以通过已有的Xcode项目转化而来

Step.2 - Pod工程开发

添加新文件有两种方式，通过Xcode直接添加，和通过文件+pod添加。
通过Xcode添加时需要注意，选择所属的project时要选择Pods而不能选择主工程。
如果项目本身是用cocoaPods创建的话，那么也可以直接将文件粘贴至/libName/Classes文件夹下，然后回到Example目录执行pod install。
所有引入的代码都会正确陈列在Xcode索引中的-> Pods -> Development目录下。
