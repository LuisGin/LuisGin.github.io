Introduction

Swift作为一门专为iOS设计的新兴编程语言，在性能、语法、可扩展性等方面都展现出了可观的能力。从诞生之初至今升级至Swift4，其语法格式已基本脱离了Objective-c所局限的内容，对于Swift的理解，也不应简单地从OC进行对照，编程过程中采取的思路甚至更多借鉴了Java和Ruby的内容。Coding应用更适应Swift的规范，也同样有助于整理逻辑和提升代码质量。
下文将从设计思路上，分段介绍Swift常用的结构，如果你对OC足够熟悉，那么你将会发现这也是Swift重构后与OC最大的变化所在。

Phrase 1 - Struct

显著变化之一是Swift广泛采用Struct替代了原本的NSObject。Struct为值引用，NSObjcet为地址引用，一个简单的例子是对Struct类型的Array赋值后地址会发生变化而NSArray则不会。
Swift本身为Struct提供了丰富的方法，如Array的append可以避免NSMutableArray和NSArray的不便。也因为Swift是类型安全的语言，Array和Dictionary必须声明类型。

Optional

有一个十分常见的语法Type?或Type!，即在类型后添加问号或者感叹号，这其实是swift的一种语法糖，即简写了一个系统Struct。模仿底层的写法将与如下的Struct完全一致。

系统额外实现了一些常用的协议用来简化解包这一操作，因此String与String?间需要安全地转化操作也就不难理解了。

Phrase 2 - Protocol

协议式编程是MVP的一种实现，将’响应动作’独立成块，从冗长的代码中区别出来。
Swift协议是一次广义的’声明’->‘实现’。

必须要关注到的一点是，协议实现依然是针对协议而言的(因此是抽象的’能力’)，不允许定义Stored Property，也不允许Override。另外，你必须在类定义时额外声明一次它实现了协议(很好理解，因为上文的范例代码2中只叙述了’如果是MyClassA，那么ProtocolA将这样运作’，必须再声明一次MyClassA满足ProtocolA)。
当然这样的写法也有很多可观的优势。

  • 用协议的默认实现来完成可选。 - 简单说，对协议自身的一次实现，会被默认地填充进任何一次协议的实现中，只要不被覆盖。
  • 实现协议的where条件完成限定。 - 可以更好地满足条件限定，比如可以
    protocol'可食用协议': '昆虫'&'无毒'&'有营养' 这一策略地好处在于可以组合式地进行限定，而在通常的对象继承中，可能必须定义一个无意义的中间类型。
  • 更清晰的就近原则(但除了默认实现外不会混合两个不同的实现)。 - 协议的实现会返回匹配的节点，好处和坏处都是不需要考虑override，需要视具体情形选用。
  • Swift4中新增了class和protocol的组合。

此处还有一个黑魔法(也可能是XcodeBug)，对于Struct而言，当定义了任意一个init函数后，默认的全参init函数会被取消，而通过Extension Protocol可以不抛错地引回该函数。

Phrase 3 - 运行时指针

swift不包含运行时指针！swift不包含运行时指针！swift不包含运行时指针！重要的事情说三遍。
很明显的一个例子是#Selector(ClassA.funcA(:))对应的方法必须包含@objc关键字，因为Selector会在运行时去内存中找(此处有疑问可以查看MethodSwizzle的资料)。在swift4之前，所有附带public或open的swift方法会自动补上@objc转换，这一策略在swift4中被取消了，当然也可以通过XCProject恢复。

无特殊情况下swift是不会补上运行时指针这一底层设计的(特别是考虑到近几年WWDC公布的资料及官方不断提速的需求)。直接影响之一是迫使swift重构了整个GCD，直接证据是OC的GCD和Swift的GCD统一Label也不会被认为是同一个(类比NSNotification和Notification同名暨同体)。

Phrase 4 - 补充关键字

  1. Any和AnyObject。后者为NSObject类型，swift只提供一种类型转换，即(as Type)语法，此处需要注意的是，大部分情况下，你都需要as？或as！来进行转换，然而语言本身以不开放地方式提供了绝大部分NS类和swift结构体之间的转换，只有这些转换是被认证安全的可以不以Optional进行判断。Any和AnyObject可以假设认为是’NSAny’和Any，两者间可以安全地downCast，前者必须继承NSObject。
  2. mutating和inout。由于结构体不允许在实例方法内改变实例本身(否则实例地址变了，却还要执行前面的方法产生了conflict)，因此引入了mutating修饰func，允许在方法中改变实例(enum也算结构体，class无需考虑)。inout可以认为是函数级的，用来修饰参数，代表传参可变并且引用了实际地址而非形式参数。
  3. open和fileprivate。补充作用域，前者可继承可重定义(public不可重定义)，后者作用当前文件。特别需要注意的是，引入fileprivate后，private的作用域被开发者吐槽了很久，在swift4中策略变更为了extension后的closure会被认为在private中。
  4. required。要求继承者必须重写该方法，常用于init函数中表示design构造器。

传说中的语法糖分界线

  1. guard { return Bool }() else { return/break/continue } - 类似于安全形的Assert断言，对于代码结构有极大地提升，主要注意的是else闭包中必须包含可以“跳出Closure”的语句，但是选用三者中的哪一种需要视需求而定。
  2. Optional(T) ?? Optional(Y) - 若空则选用后者，T与Y不要求是同一模版，"??"语法糖前的量必须是可选值。
  3. if let { opt } - 完全等价于 { let a; if a != nil { opt } }，有同义的guard let/if var语法，常用于正向的条件判断。

Phrase 5 - 常用需求与设计

1.单例模式。

  • 直接let或var定义，只需要考虑定义域，在最外层即全局单例。
  • 用init来返回同一instance，或直接通过private／fatelError／制定构造器来屏蔽init函数。建议在编译阶段抛出错误。

2.容错处理。

  • Optional和Try_Catech都可以实现容错
  • 可忽视的用Optional，不可忽视的用try_Catch。强烈建议先考虑try_Catch，后者才是符合错误统一处理设计理念。
